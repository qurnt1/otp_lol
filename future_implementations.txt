FUTURES IMPLEMENTATIONS
======================

PRIORITES PRINCIPALES
--------------------
1. Refaire l'historique des actions pour qu'il soit vraiment lisible par un humain.
2. Ajouter le choix du site de stats prefere pour le bouton principal.
3. Ajouter des toggles rapides directement sur l'interface principale.
4. Finaliser l'aperçu Pick / Ban / Sorts pour qu'il soit plus propre visuellement.
5. Ajouter un vrai systeme de theme clair / sombre.
6. Preparer une release v7 coherente partout.

1. HISTORIQUE DES ACTIONS
-------------------------
- Refaire l'affichage de l'historique des actions.
- Rendre l'historique plus lisible, plus propre et moins brut.
- Eviter le rendu trop proche d'un JSON.
- Normaliser les messages affiches dans tout le flux.
- Mieux formuler les evenements importants :
- auto-accept
- hover
- ban
- pick
- sorts
- reconnexion
- erreurs
- Afficher une ligne claire par evenement.
- Afficher une heure facile a lire.
- Avoir un type d'evenement plus propre : info, succes, erreur, etc.
- Rendre l'ensemble plus agreable a scanner visuellement.

2. SITE DE STATS PREFERE
------------------------
- Ajouter un parametre "site de stats prefere".
- Proposer plusieurs choix :
- OP.GG
- DeepLOL
- Porofessor
- League of Graphs
- Sauvegarder ce choix dans la configuration.
- Faire en sorte que le bouton principal ouvre automatiquement le site choisi.
- Adapter le texte du bouton principal si besoin.
- Eventuellement afficher le logo du site choisi sur le bouton principal.

3. TOGGLES RAPIDES SUR L'INTERFACE PRINCIPALE
---------------------------------------------
- Permettre d'activer ou de desactiver directement les options depuis l'interface principale.
- Cliquer sur la zone "Pick" pour activer / desactiver l'auto-pick.
- Cliquer sur la zone "Ban" pour activer / desactiver l'auto-ban.
- Cliquer sur la zone "Sorts" pour activer / desactiver l'auto-summoners.
- Garder un retour visuel immediat sur l'etat ON / OFF.
- Verifier que ces clics ne cassent pas les autres interactions de l'interface.

4. APERCU PICK / BAN / SORTS
----------------------------
- Faire en sorte que l'aperçu soit plus propre quand une option est desactivee.
- Quand une section est OFF, ne pas afficher plusieurs "OFF" a la place des images.
- Garder uniquement l'etat global ON / OFF pour la section.
- Masquer les images si la section est desactivee.
- Conserver une hauteur stable et un alignement propre meme quand les images sont masquees.
- Continuer a peaufiner l'espacement et l'alignement visuel sur l'interface principale.

5. THEMES ET APPARENCE
----------------------
- Ajouter un choix de theme pour l'application.
- Proposer au minimum un theme sombre et un theme clair.
- Ajouter ce choix dans les parametres.
- Sauvegarder le theme choisi dans la configuration.
- Appliquer le theme sans casser le style actuel de l'application.
- Si possible, permettre un changement de theme en live sans redemarrage.

6. EXPORT DE CONFIG
-------------------
- Ameliorer le bouton "Exporter config".
- Garder l'ouverture de l'explorateur de fichiers.
- Pre-remplir automatiquement un nom de fichier par defaut.
- Exemples de noms possibles :
- mainlol_config.json
- mainlol_config_backup.json
- mainlol_config_YYYY-MM-DD.json
- Eviter que l'utilisateur doive taper le nom du fichier a la main a chaque export.

7. PARAMETRES A COMPACTER
-------------------------
- Retirer encore un peu de hauteur a la fenetre des parametres.
- Reduire l'espace noir restant en bas.
- Garder une fenetre dense mais toujours lisible.
- Verifier qu'il n'y ait pas besoin d'un scroll inutile.

8. STYLE VISUEL DES BOUTONS
---------------------------
- Rendre les boutons plus modernes.
- Arrondir davantage les coins et les bordures si le toolkit le permet.
- Harmoniser le rendu entre :
- la fenetre principale
- la fenetre des parametres
- les popups

9. PREPARATION RELEASE V7
-------------------------
- Passer toute l'application en version 7 avant la prochaine release.
- Verifier la coherence de la version partout.
- Controler avant release :
- version dans le code
- version dans le README
- version dans le script de build
- version dans les fichiers de config ou metadata si besoin
- version dans les releases GitHub si utilisees
- Faire une vraie passe de verification generale avant publication.
- Eviter toute incoherence entre l'application, la doc et le build.
